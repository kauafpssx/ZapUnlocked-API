## <Icon icon="triangle-exclamation" iconType="sharp-regular" color="#9d4fe6" size={22} /> Reportar um problema

Use as **Issues do GitHub** para reportar bugs ou solicitar funcionalidades.

<Steps>
  <Step title="Verifique se já existe" icon="magnifying-glass">
    Antes de abrir, pesquise nas [issues existentes](https://github.com/kauafpssx/ZapUnlocked-API/issues) se o problema já foi reportado ou está sendo tratado.
  </Step>
  <Step title="Escolha o template correto" icon="file-lines">
    Ao abrir uma issue, selecione o template adequado: **Bug Report** para erros ou **Feature Request** para sugestões.
  </Step>
  <Step title="Preencha com detalhes" icon="pen-to-square">
    Inclua versão do sistema, passos para reproduzir, comportamento esperado vs. atual e logs relevantes. Issues vagas demoram mais para serem resolvidas.
  </Step>
</Steps>

<CardGroup cols={2}>
  <Card title="Abrir Bug Report" icon="bug" href="https://github.com/kauafpssx/ZapUnlocked-API/issues/new?template=bug_report.md">
    Algo não funciona como esperado.
  </Card>
  <Card title="Abrir Feature Request" icon="lightbulb" href="https://github.com/kauafpssx/ZapUnlocked-API/issues/new?template=feature_request.md">
    Sugestão de nova funcionalidade ou melhoria.
  </Card>
</CardGroup>

## <Icon icon="code-pull-request" iconType="sharp-regular" color="#9d4fe6" size={22} /> Contribuir com código

<Steps>
  <Step title="Fork e clone" icon="code-fork">
    Faça fork do repositório e clone localmente:
    ```bash
    git clone https://github.com/SEU_USER/ZapUnlocked-API.git
    cd ZapUnlocked-API
    ```
  </Step>
  <Step title="Crie uma branch" icon="code-branch">
    Use nomes descritivos seguindo o padrão:
    ```bash
    git checkout -b feat/nome-da-feature
    # ou
    git checkout -b fix/nome-do-bug
    ```
  </Step>
  <Step title="Faça as alterações" icon="code">
    Implemente sua mudança. Mantenha o escopo focado: um PR por feature ou fix.
  </Step>
  <Step title="Commit seguindo o padrão" icon="check">
    Use [Conventional Commits](https://www.conventionalcommits.org/):
    ```
    feat: adiciona endpoint de envio de localização
    fix: corrige timeout na fila de mídia
    docs: atualiza exemplos de webhook
    refactor: extrai lógica de auth para módulo separado
    ```
  </Step>
  <Step title="Abra o Pull Request" icon="code-pull-request">
    Abra o PR para a branch `main` preenchendo o template. Descreva o que muda, por que e como testar.
  </Step>
</Steps>

## <Icon icon="scale-balanced" iconType="sharp-regular" color="#9d4fe6" size={22} /> Padrões do projeto

<AccordionGroup>
  <Accordion title="Estilo de código" icon="pen-nib">
    - Python: siga o PEP 8. Use nomes descritivos em inglês para variáveis e funções.
    - Sem código comentado deixado para trás. Sem `print()` de debug em PRs.
    - Imports organizados: stdlib, terceiros, locais.
  </Accordion>
  <Accordion title="Commits" icon="git-alt">
    Padrão **Conventional Commits** obrigatório:

    | Prefixo | Quando usar |
    |---|---|
    | `feat:` | Nova funcionalidade |
    | `fix:` | Correção de bug |
    | `docs:` | Documentação |
    | `refactor:` | Refatoração sem mudança de comportamento |
    | `chore:` | Tarefas de manutenção (deps, configs) |
    | `test:` | Testes |
  </Accordion>
  <Accordion title="Pull Requests" icon="code-pull-request">
    - Um PR por feature ou fix. PRs grandes dificultam review.
    - Título no mesmo padrão dos commits: `feat: ...`, `fix: ...`
    - Preencha o template completamente. PRs sem descrição serão fechados.
    - PRs abertos para branches erradas serão redirecionados.
  </Accordion>
  <Accordion title="Segurança" icon="shield-halved">
    Nunca abra uma issue pública para reportar vulnerabilidades de segurança. Leia a [Security Policy](https://github.com/kauafpssx/ZapUnlocked-API/security/policy) e reporte de forma privada via **GitHub Security Advisories**.
  </Accordion>
</AccordionGroup>
